1、Hive 表关联查询，如何解决数据倾斜问题？

排查思路：

定位查询慢的sql，统计key，count(*)；统计null值情况，主要确定key是否分布均衡.

情况一：group by 导致 key 分布不均；count distinct语法

map端实现部分聚合操作，效率更高但需要更多内存；

生成两个查询计划MRJob；第一个将Map结果随机分布到 Reduce 中，达到负载均衡，每个Reduce再聚合操作并输出结果。（rsq：这个局限性？）第二个MRJob根据上面结果，再按照GroupBY Key分不到 Reduce中（保证相同key分布到Reduce中，因为已经两次聚合过，此时数据量应该很小），这样就保证整体group by 没问题，相同key分到一个reduce中。

局限性：第二个参数不支持多个 distinct 列，不能提前进行聚合处理

异常情况：https://blog.csdn.net/Dreamy_zsy/article/details/112679598

todo: 如果key本身倾斜, 是否可以通过取模随机化呢？参数底层是否如此，先取模计算一次，数据量变小再聚合；

todo: sparkSQL是否已优化？

对于大数据量且key倾斜的 count(distinct xx) 情况，会导致数据分布到一个Reduce Task上；修改为group by 语法。新版Hive3.x已自动优化，详情：https://www.modb.pro/db/81149

情况二：大小表 join

通过 map join 将小表先存入内存

场景1：间接实现mapjoin，通过限制行和列，对于本案例就是过滤没有成交的卖家。再使用 mapjoin

场景2：倾斜值确定，比如null值；核心是将倾斜值随机分发到 Reduce，通过join 时对这些特殊值 concat 随机数

由于倾斜的卖家大量存在而且动态变化, 很少会有使用场景；

3、Hive 中 Sort By, Order By, Cluster By, Distrbute By 使用场景

order by: 全局排序。数据量大不使用，只有一个reduce输出文件

sort by: 进入reducer 前排序，仅保证每个reducer输出有序，形成多个有序重叠范围的排序文件。

distribute by: 根据指定字段分发到 reducer；常结合 sort by 使用，且hive要求 distribute by 要在 sort by 之前；形成多个非重叠未排序文件

cluster by: sort by + distribute by；只能倒序排序。形成多个非重叠排序文件

场景：

  • 数据规模较大时，order by 不适用，用 distribute by + sort by

    原理：类似归并(分治)排序，先局部排序，再全局排序。

  • 当排序指定同一个时，cluster by = distribute by + sort by

参考：https://stackoverflow.com/questions/13715044/hive-cluster-by-vs-order-by-vs-sort-by

6、HQL转换为MR过程

简化：

  1. HQL 解析成 TaskTree（SQL Parser ，Physical Plan，Query Optimizer，Execution）

扩展：

HiveSQL -> AST(抽象语法树) -> QB(查询块) -> OperatorTree(操作树) -> 优化后的操作树 -> mapreduce任务树 -> 优化后的 mapreduce 任务树

Parser(将 HQL 转换成抽象语法树) -> Semantic Analyzer(将抽象语法树转换成查询块) -> Logic Plan Generator(查询块转逻辑查询计划) -> LogicalOptimizer(重写逻辑查询计划) -> Physical Plan Generator(将逻辑计划转换成物理计划 M/R jobs)

SQL Parser：Antlr定义SQL的语法规则，完成SQL词法，语法解析，将SQL转化成抽象语法树AST Tree

Semantic Analyzer: 遍历AST Tree, 抽象出查询的基本组成单元QueryBlock

Logical plan: 遍历QueryBlock，翻译为执行操作树OperatorTree

Logical plan optimizer: 逻辑层优化器进行OperatorTree变换，合并不必要的ReduceSinkOperator，减少shuffle数据量

Physical plan: 遍历OperatorTree，翻译为MR任务

Logical plan optimizer: 物理层优化器进行MR任务的变换，生成执行计划。

MR过程：分片、Map、shuffle、reduce

源码层分析转换过程：https://blog.csdn.net/qq_32727095/article/details/122112266
